# Collector в sysbeat: подробное руководство

## 1. Что такое Collector и зачем он нужен

**Collector** — это скрипт (программа), которая запускается на Linux-устройстве и **собирает системные метрики**: сколько загружен процессор, сколько занято памяти, какая нагрузка на систему. Затем эти данные отправляются на сервер.

### Аналогия

Представь, что у тебя есть умный дом. В каждой комнате стоит датчик температуры. **Collector** — это как раз этот датчик: он смотрит на показания (читает `/proc`) и отправляет их на центральный пульт (сервер). Сервер, в свою очередь, показывает эти данные на экране (dashboard) через WebSocket.

### Три участника системы

```
┌───────────────┐      HTTP POST       ┌───────────────┐      WebSocket       ┌───────────────┐
│   Collector   │ ──────────────────>  │    Server     │ ──────────────────>  │   Dashboard   │
│  (на устрой-  │    вот мои метрики   │  (получает,   │    новые данные!     │  (браузер,    │
│   стве Linux) │                      │   хранит,     │                      │  видит график)│
└───────────────┘                      │   рассылает)  │                      └───────────────┘
                                       └───────────────┘
```

Collector **не использует WebSocket напрямую** — он шлёт обычные HTTP-запросы. Но он является **источником данных** для всей системы, включая WebSocket-рассылку на dashboard.

## 2. Откуда Collector берёт данные

В Linux есть специальная виртуальная файловая система `/proc` — это не настоящие файлы на диске, а "окна" в ядро операционной системы. Через них можно узнать, что происходит в системе прямо сейчас.

### 2.1. `/proc/stat` — загрузка процессора

Пример содержимого:

```
cpu  2255 34 2290 25563 6290 127 456 0 0 0
cpu0 1132 17 1145 12781 3145 63 228 0 0 0
intr 1234567 ...
```

Коллектор читает **первую строку**, начинающуюся с `cpu ` (с пробелом!). Числа означают время (в тиках процессора), которое система потратила на:

| Поле | Что означает |
|------|-------------|
| 1 (user) | Обычные пользовательские программы |
| 2 (nice) | Программы с пониженным приоритетом |
| 3 (system) | Работа ядра (драйверы, системные вызовы) |
| 4 (idle) | Процессор простаивает |
| 5 (iowait) | Ожидание диска |
| 6 (irq) | Аппаратные прерывания |
| 7 (softirq) | Программные прерывания |
| 8 (steal) | Виртуализация (если система внутри VM) |

**Как считается загрузка CPU:**

Нельзя просто взять одно значение из `/proc/stat` и сказать "загрузка 45%". Нужно **сравнить два замера во времени**:

```
Замер 1 (сейчас):     user=1000, system=500, idle=8000
Замер 2 (через 1с):   user=1050, system=520, idle=8430

Разница:               user=+50,  system=+20, idle=+430
Всего тиков: 50 + 20 + 430 = 500

usage = (50 + 20) / 500 * 100 = 14%   <-- загружен на 14%
idle  = 430 / 500 * 100 = 86%          <-- простаивает 86%
```

Это и делает Collector: он запоминает предыдущие значения и при следующем замере считает разницу.

### 2.2. `/proc/meminfo` — память

Пример:

```
MemTotal:       16384000 kB
MemFree:         2048000 kB
MemAvailable:    8192000 kB
Buffers:          512000 kB
Cached:          4096000 kB
```

Всё в **килобайтах** (kB). Collector делит на 1024, чтобы получить мегабайты.

| Поле | Что означает |
|------|-------------|
| MemTotal | Вся оперативная память |
| MemAvailable | Сколько доступно прямо сейчас (без очистки кэша) |
| MemFree | Полностью свободная (не используется вообще) |

**Формулы:**
```
used = total - available
percent = used / total * 100
free = MemFree (полностью незанятая память)
```

**Важное различие:**
- `memory.free` в payload = `MemFree` — память, которая вообще не используется.
- `memory.used` считается как `total - available` (где `available` включает кэш и буферы, которые можно освободить).
- Таким образом `free + used ≠ total` — это нормально, потому что часть памяти занята кэшем файловой системы.

Если `MemAvailable` нет (старые Linux), используем `MemFree + Buffers + Cached` для расчёта `used`.

### 2.3. `/proc/loadavg` — нагрузка

Пример:

```
0.25 0.15 0.10 2/345 12345
```

Первые три числа — это **среднее количество процессов**, которые ждут процессор:
- `0.25` — за последнюю минуту
- `0.15` — за последние 5 минут
- `0.10` — за последние 15 минут

Значение `1.0` означает, что процессор загружен на 100% (все ядра заняты). Если у тебя 4 ядра, то `4.0` — это полная загрузка.

## 3. Как Collector парсит /proc (разбор parser.ts)

### 3.1. Чтение файла

```typescript
import { readFileSync } from 'fs';

const strContent = readFileSync('/proc/stat', 'utf-8');
```

`readFileSync` — синхронное чтение. Это нормально, потому что `/proc` — виртуальная ФС, файл читается мгновенно из памяти ядра. Никакого обращения к диску.

### 3.2. Парсинг CPU

```typescript
const strCpuLine = strContent.split('\n').find((strLine) => strLine.startsWith('cpu '));
```

Ищем строку, начинающуюся с `cpu ` (с пробелом). Важно: `cpu0`, `cpu1` и т.д. — это отдельные ядра, а `cpu ` (общая статистика).

```typescript
const arrFields = strCpuLine.trim().split(/\s+/).slice(1).map(Number);
```

- `trim()` — убираем лишние пробелы
- `split(/\s+/)` — разбиваем по пробелам (один или несколько)
- `slice(1)` — пропускаем слово `cpu`
- `map(Number)` — превращаем строки в числа

### 3.3. Хранение предыдущих значений

```typescript
let nPrevUser = 0;
let nPrevSystem = 0;
let nPrevIdle = 0;
let bFirstRun = true;
```

Эти переменные живут **вне функции** — на уровне модуля. Они сохраняются между вызовами. Это важно: при первом запуске не с чем сравнивать, поэтому возвращаем `usage: 0`.

### 3.4. Расчёт дельты

```typescript
if (bFirstRun) {
  bFirstRun = false;
  nPrevUser = nUser;
  // ... сохраняем все значения
  return { nUsage: 0, nUser: 0, nSystem: 0, nIdle: 100 };
}

const nDeltaTotal = nTotal - nPrevTotal;
const nUsage = 100 * (nDeltaUser + nDeltaSystem + nDeltaIrq + nDeltaSoftirq) / nDeltaTotal;
```

В формулу входят:
- `user` — пользовательские программы
- `system` — ядро
- `irq` и `softirq` — прерывания

Не входят:
- `idle` — простой
- `iowait` — ожидание диска (считается неактивностью)
- `nice` — пониженный приоритет (можно включить или исключить)

### 3.5. Парсинг памяти

```typescript
for (const strLine of strContent.split('\n')) {
  const arrMatch = strLine.match(/^([A-Za-z()]+):\s+(\d+)\s+kB/);
  if (arrMatch) {
    mapValues.set(arrMatch[1], Number(arrMatch[2]));
  }
}
```

Используем **регулярное выражение**:
- `^` — начало строки
- `([A-Za-z()]+)` — название (например, `MemTotal`)
- `:\s+` — двоеточие и пробелы
- `(\d+)` — число
- `\s+kB` — пробелы и `kB`

Результат складываем в `Map` — удобный словарь, где по имени можно быстро получить значение.

## 4. Отправка на сервер (разбор sender.ts)

### 4.1. Почему без внешних библиотек

В Node.js встроены модули `http` и `https`. Они позволяют делать запросы без `axios` или `node-fetch`. Это:
- Меньше зависимостей (нет проблем с обновлениями)
- Меньше размер (важно для embedded-устройств)
- Понимание "как это работает под капотом"

### 4.2. HTTP-запрос "вручную"

```typescript
import { request } from 'https';
import { request as requestHttp } from 'http';

const fnRequest = objUrl.protocol === 'https:' ? request : requestHttp;

const objReq = fnRequest(objRequestOptions, (objRes) => {
  // обработка ответа
});

objReq.write(strBody);
objReq.end();
```

**Как это работает:**

1. Создаём объект запроса `objReq` через `request(options, callback)`
2. `callback` вызывается, когда сервер прислал **заголовки** ответа
3. `objRes.on('data', ...)` — читаем тело ответа по кусочкам
4. `objRes.on('end', ...)` — ответ полностью получен
5. `objReq.write(strBody)` — отправляем тело запроса
6. `objReq.end()` — завершаем запрос

Это **асинхронно**: `request()` не блокирует программу. Node.js продолжает работать, а когда ответ придёт — вызовет callback.

### 4.3. Retry с экспоненциальным backoff

Что делать, если сервер недоступен? Нельзя просто падать с ошибкой — устройство может временно потерять связь.

```typescript
while (true) {
  nAttempt++;
  const objResult = await trySend(objPayload);

  if (objResult.bSuccess) return objResult;

  const nWait = Math.min(nBackoffMs + nJitter, nMaxBackoffMs);
  await sleep(nWait);
  nBackoffMs = Math.min(nBackoffMs * 2, nMaxBackoffMs);
}
```

**Принцип:**

| Попытка | Задержка |
|---------|----------|
| 1 | 1 сек + jitter |
| 2 | 2 сек + jitter |
| 3 | 4 сек + jitter |
| 4 | 8 сек + jitter |
| 5+ | 30 сек (максимум) |

**Jitter** — случайное число 0–200 мс. Нужно, чтобы если 1000 устройств одновременно потеряли связь, они не начали бомбить сервер **одновременно** после восстановления.

### 4.4. Timeout

```typescript
objReq.on('timeout', () => {
  objReq.destroy();  // <-- принудительно закрываем соединение
});
```

Если сервер не отвечает 5 секунд, соединение обрывается. Без этого программа могла бы ждать вечно.

## 5. Главный цикл (разбор index.ts)

### 5.1. Архитектура

```
┌────────────┐      ┌────────────┐      ┌────────────┐      ┌────────────┐
│ setInterval│ ───> │ collect()  │ ───> │ parse /proc│ ───> │ send HTTP  │
│  (каждую   │      │            │      │            │      │            │
│   секунду) │      │            │      │            │      │            │
└────────────┘      └────────────┘      └────────────┘      └────────────┘
```

### 5.2. Защита от "наложения"

```typescript
let bSending = false;

async function collectAndSend(): Promise<void> {
  if (!bRunning) return;
  if (bSending) return;  // <-- пропускаем, если предыдущий ещё не завершился

  bSending = true;
  const objResult = await sendMetrics(objPayload);
  bSending = false;
}
```

Если отправка занимает больше секунды (например, из-за retry), а `setInterval` сработал снова — не начинаем новую отправку. Это защита от "наслоения" запросов.

### 5.3. Graceful shutdown

```typescript
process.on('SIGINT', () => void gracefulShutdown());
process.on('SIGTERM', () => void gracefulShutdown());

async function gracefulShutdown(): Promise<void> {
  bRunning = false;
  clearInterval(timerInterval);

  if (bSending) {
    // Ждём завершения текущей отправки
    while (bSending && nWaited < 10000) {
      await sleep(100);
      nWaited += 100;
    }
  }

  process.exit(0);
}
```

**SIGINT** — сигнал прерывания (Ctrl+C в терминале).
**SIGTERM** — сигнал "пожалуйста, завершись" (от systemd при остановке сервиса).

Коллектор:
1. Останавливает таймер (новые замеры не начинаются)
2. Ждёт завершения текущей отправки (до 10 секунд)
3. Корректно завершает процесс

Это важно: если просто убить процесс (`kill -9`), может потеряться последняя метрика.

### 5.4. Логирование

```
[2026-05-09T21:50:27.558Z] cpu: 3.3% mem: 5.6% load: 0.46 status: 200
```

- `[` + ISO timestamp + `]` — когда произошло
- `cpu: 3.3%` — загрузка процессора
- `mem: 5.6%` — занято памяти
- `load: 0.46` — нагрузка за минуту
- `status: 200` — HTTP-ответ сервера (или `failed` / `retry`)

## 6. Как Collector связан с WebSocket

Хотя Collector сам не использует WebSocket, он является **началом цепочки**, которая заканчивается WebSocket-рассылкой.

### Полный flow данных

```
1. Collector читает /proc/stat, /proc/meminfo, /proc/loadavg
   |
   | (каждые INTERVAL_MS, например 1 сек)
   v
2. Collector формирует JSON: { deviceId, timestamp, cpu, memory, load }
   |
   | HTTP POST /ingest
   | Authorization: Bearer <token>
   v
3. Сервер (Fastify) получает запрос:
   - Проверяет токен
   - Проверяет rate limit
   - Сохраняет в SQLite
   |
   | <-- внутри сервера --
   v
4. Сервер вызывает broadcastUpdate(metric)
   |
   | WebSocket message: { type: "update", deviceId, metric }
   v
5. Dashboard (браузер) получает сообщение
   |
   v
6. График на экране обновляется в реальном времени
```

**Ключевой момент:** WebSocket даёт **мгновенное** обновление dashboard. Без него браузеру пришлось бы раз в секунду спрашивать сервер "а есть что новое?" (polling), что создаёт лишнюю нагрузку.

Collector же просто "кормит" сервер данными. А сервер решает, как их доставить клиентам.

## 7. Зачем проект учебный

Этот collector создан, чтобы на практике понять:

1. **Как читать системную информацию в Linux** — через `/proc`, без внешних утилит
2. **Как работает HTTP в Node.js на низком уровне** — через `http.request`, без обёрток
3. **Как писать отказоустойчивый код** — retry, backoff, graceful shutdown
4. **Как связать несколько компонентов** — collector, server, dashboard

В реальном продакшене можно было бы:
- Использовать `axios` или `fetch` вместо ручного HTTP
- Взять библиотеку вроде `systeminformation` вместо парсинга `/proc`
- Добавить буферизацию (хранить метрики локально при отсутствии сети)

Но здесь цель — **понять механизм**, а не сделать максимально быстро.
